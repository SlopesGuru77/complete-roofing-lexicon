### Root cause

`zbpack.json` â†’ `"output_dir": "public"` is **only applied when there is a `build_command`**. For pure-static no-build plans (this repo), zbpack's generated Dockerfile is:

```dockerfile
FROM alpine AS output
COPY . /                                  # <-- entire repo
FROM docker.io/zeabur/caddy-static:latest
COPY --from=output / /usr/share/caddy
```

Confirmed in the build log: `#6 [output 1/1] COPY . /`. `output_dir` is silently ignored for this code path. (Source: zbpack docs say `output_dir` is "where built files go" â€” implicitly assumes there's a build step that produces them.)

## 4. The lesson worth keeping

**`zbpack.json` â†’ `output_dir` is a no-op for pure-static no-build deploys on Zeabur.** If a real fix for the file-exposure issue is wanted, the options are:

1. **Add a `build_command` that produces a curated dir.** e.g. `"build_command": "mkdir -p dist && cp index.html sw.js manifest.json og-cover.png icon-*.png dist/"` plus `"output_dir": "dist"`. Pros: single-line config, no source-tree restructuring. Cons: violates the "no build step" hard rule in CLAUDE.md and the v2 handoff Â§3.2 â€” though arguably a file-copy is not a "build" in the spirit of that rule, only in its letter. Owner call.
2. **Try `"app_dir": "public"`** with files moved into `public/`. Untested. The semantics: `app_dir` tells zbpack to *analyze* a subdir as the project root. Plausibly makes `COPY . /` resolve from `public/`, which would give us the desired behavior. This was the natural next step but I declined to attempt it after the first failed deploy because (a) site was already down, (b) user wasn't available, (c) another failed attempt risked a longer outage. Worth testing in a future session **after** demonstrating it in a Zeabur preview env or with a dry-run.
3. **Custom `Dockerfile`** in the repo. Overrides zbpack's auto-generation. Maximum control. Cons: now you maintain a Dockerfile, which the v2 spec said to avoid.
4. **Accept the exposure.** The repo is public on GitHub already â€” none of the leaked files (CLAUDE.md, HANDOFF-*.md, supabase-setup.sql, tests/, package.json) are secret. The aesthetic of "polished static deploy ships only the shell" is real but the security/legal risk is zero. The dirs that *would* matter (`_staging/`, `node_modules/`, `Building Owners Guide/`, `*.log`) are already gitignored and never reach the deploy.

My recommendation: **option 4 for now** (do nothing â€” the exposure is cosmetic, not material), with **option 2 (`app_dir: "public"`)** as the lowest-risk fix to try in a future session if cosmetics start to matter. Avoid options 1 and 3 unless the rule against build steps is consciously relaxed.
