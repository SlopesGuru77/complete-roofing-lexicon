### Features

**1. Term Verification Workflow (`feat(vf)`)**
- `state.vf = {[termKey]: {date, by}}` tracks per-term verification with a 365-day freshness window (`VF_STALE_DAYS`).
- `vfStatus(termKey)` returns `'fresh' | 'stale' | 'never'`; `markVerified(termKey, by)` / `unmarkVerified(termKey)` mutate state and save.
- Lexicon term cards swap among three badge states: gold `Verify Current Spec` (default for vf-flagged), green `Verified YYYY-MM` (fresh, tooltip carries full date + verifier), red `Verification Stale` (>365d).
- Manager dashboard gains a **Term Verification Queue** panel between Cert History and Reports & Transfer. Sorted stale-first, then never, then fresh. Default filter shows only the work queue; toggle reveals all 30 flagged terms. Per-row "Verify" button stamps today's date; "Mark all visible verified today" covers post-annual-review bulk action.
- Helpers exposed on `window.__crl`: `vfStatus`, `markVerified`, `unmarkVerified`, `VF_STALE_DAYS`, `vfState` (getter).
- Plausible events: `Term Verified`, `Verification Bulk Marked`.
- New spec: `tests/term-verification.spec.js` (6 tests).

**2. Team Roll-Up via JSON Import (`feat(manager)`)**
- The existing `supabase-setup.sql` schema intentionally does **not** grant SELECT to the anon role (write-only by design), so a Supabase READ path for team roll-up needs a schema change + auth-model decision the user hasn't committed to. This commit delivers the team-value *now* via an offline JSON-import flow that's forward-compatible with eventual Supabase reads.
- `crl_team_v1` localStorage key holds imported per-device snapshots (separate from per-device state).
- Helpers: `teamRead`, `teamImport(json|string)`, `teamRemove(id)`, `teamClear()`, `teamCombinedCerts()`, `TEAM_KEY` — all on `__crl`.
- Dedupe by `name|role|location` slug — same staff re-imported overwrites the previous snapshot (intended; latest export is canonical). `TEAM_MAX_DEVICES = 50`.
- Manager → **Team Roll-Up (Import)** section: multi-file JSON file input, imported-devices table with per-row Remove, "Clear All Imports" (disabled when empty), Combined Certifications table merging local + imported certs sorted by date desc with a "source" column distinguishing them.
- `teamImport` rejects: malformed JSON, non-CRL objects (missing `v`/`cards`/`certs`), arrays, source snapshots with no staff name. Returns `{ok, msg}` — caller surfaces the message in the panel status line.
- Plausible events: `Team Import`, `Team Device Removed`, `Team Imports Cleared`.
- New spec: `tests/team-import.spec.js` (7 tests).

### Bugs fixed this session

**Zeabur deploy regression** (commit `e6f7982`).

- This session's first push (commit `f7d7bdf`) deployed but the Zeabur Nixpacks auto-detector flipped `plan_type` from `static` → `nodejs`. It baked `CMD ["node", "/src/index.js"]` into the image and the pod crash-looped with `MODULE_NOT_FOUND`. Live host returned 502 Bad Gateway.
- Previous deploys (commits before `f7d7bdf`) all reported `PLANTYPE = static` per `zeabur deployment list`. The detection appears non-deterministic when `package.json` is present, lacks a `main` field, and there's no language hint.
- Fix: committed `zbpack.json` with `{"plan_type": "static", "output_dir": "."}`. This pins the static-files plan and serves the repo root with no Node runtime, no install step, no entry-point guessing. Schema: `https://schema.zeabur.app/zbpack.json`.
- Webhook from GitHub to Zeabur lagged by several minutes after the fix push, so I ran `npx zeabur@latest deploy --project-id ... --service-id ... --interactive=false` to force the rebuild immediately. New build: `PLANTYPE = static`, status `RUNNING`, live host returning 200 within ~90 s.

## 5. Known issues surfaced by smoke (not blockers)

- **Plausible CORS errors.** Every `track()` call against `https://plausible.io/api/event` returns CORS-blocked. The bulk-mark action surfaced 20 of these in one click (1 per term × 14 fresh events). The site/domain isn't registered in the owner's Plausible.io managed instance, OR the wrong instance URL is configured. `track()` is wrapped in try/catch so app function is unaffected — but no analytics are being collected from production right now. Owner action — same Plausible follow-up from the previous handoff ("configure Goals"); registering the site domain in Plausible should fix both this and Goals at once.
- **Static deploy root exposes test-only files.** With `plan_type=static, output_dir=.`, the served root includes `tests/`, `tools/`, `*.log`, `node_modules/`, the `_staging/` parking lot, and the `Building Owners Guide to Commercial Roofing/` source folder. Not a *new* exposure (repo is public on GitHub), but a polished static deploy should ship only the shell. Mitigations:
  1. Move test/tooling into `dev-only/` subdir and update relative paths.
  2. Or add a build step that copies just the deploy artifacts (HTML, sw, manifest, icons, og-cover) into a curated dir referenced by `output_dir` — but that violates the "no build" hard rule.
  3. Lowest-effort: add a `.zeaburignore` if Zeabur supports one, or move the leaky dirs into git-ignored locations.
